# 멘로그(Menlog) 기획서 v2 — 4탭 + 아이콘 구조

> 작성일: 2026-07-13
> 배경: PDF 목업(1a "구움빛 다이어리" 스타일) 채택 + 하단 탭 구조(지도/피드/기록하기/추천)로 확장
> '친구'는 사용 빈도가 낮은 기능으로 판단해 탭이 아닌 지도 화면 상단 아이콘으로 배치 변경

## 1. 개요

- 기존에는 "홈(지도 단일 화면) → 기록 폼 → 초대 화면"의 단선적 흐름이었으나, 이번 기획부터 **하단 탭 4개(지도/피드/기록하기/추천) + 지도 화면 상단 아이콘(친구/그룹 관리)** 구조로 전환
- 디자인은 PDF 1a 스타일(세리프 타이틀, 폴라로이드 사진 핀) 채택 — 기존 확정 디자인 토큰(Primary `#C8600A`, Georgia 세리프 워드마크)과 정합성 확인 완료
- **탭바 재배치 판단 근거**: 지도·피드·기록하기·추천은 반복 사용 빈도가 높은 핵심 동선인 반면, 친구(그룹 관리)는 초대 시 1회성으로 쓰이고 이후엔 저빈도로 사용됨. 저빈도 기능에 탭 슬롯을 배정하면 핵심 탭들의 터치 영역만 좁아지는 손해가 있어, 지도 화면 상단 아이콘(그룹원 아바타/`ti-users`)으로 배치
- '친구' 기능은 신규 개념이 아니라 **기존 `groups`/`group_members`의 UI 라벨링 변경**이며, 배치만 탭→아이콘으로 이동
- '피드' 탭은 신규 기능이며, 좋아요/댓글 상호작용을 포함하므로 **스키마 확장 필요**

## 1.1 그룹 모델 변경 — "혼자서도 쓸 수 있는 앱" 반영

기존에는 "그룹에 소속돼야만 앱을 쓸 수 있다"는 전제였으나, 혼자 쓰는 유저도 지원하기로 결정했습니다.

- **가입 즉시 "개인 그룹" 자동 생성**: 신규 유저는 닉네임 설정 직후 백그라운드에서 본인만 속한 그룹이 자동 생성됩니다. 화면상 별도 "그룹 만들기" 단계는 없습니다 — 사용자에게는 그냥 "내 기록"으로 보입니다.
- **다중 그룹 모델 채택**: 이후 여러 친구 그룹의 초대코드를 받아 동시에 여러 그룹에 소속될 수 있습니다 (개인 그룹 + 친구 그룹 N개). 그룹 하나를 폐기하거나 갈아타는 개념이 아닙니다.
- **"전역 활성 그룹" 상태를 두지 않기로 결정** (그룹 전환 UI 방식 기각): 화면마다 그룹을 다루는 방식이 다릅니다.
  - **지도**: 내가 속한 모든 그룹(개인 포함)의 기록을 **한 지도에 병합**해서 보여주고, 그룹마다 다른 마커 색으로 구분 — 전환 없이 항상 전체를 봄
  - **피드**: 그룹마다 성격이 다른 사교 공간이라 병합하지 않고, **그룹별 서브탭**으로 분리해서 보여줌 (선택 상태는 로컬 UI 상태로만 기억, DB 저장 불필요)
  - **기록하기(+)**: 저장 시점마다 "이 기록을 어디에 남길지"(개인/그룹A/그룹B)를 명시적으로 선택 — 전역 상태에 의존하지 않음
  - **추천**: 그룹과 무관하게 유저 개인의 전체 방문 기록(모든 그룹 합산)을 취향 계산에 사용 — 기존 설계 그대로 유지
- 이 방식 덕분에 `users.active_group_id` 같은 전역 상태 컬럼이 **필요 없어졌습니다.**

## 2. 정보 구조 (IA)

| 위치 | 목적 | 데이터 소스 | 신규 스키마 |
|---|---|---|---|
| 지도 (탭) | 소속된 모든 그룹의 방문 매장을 그룹별 색상의 폴라로이드 핀으로 병합 표시 + 전국 라멘집 검색 | `ramen_shops` + `visits` + `group_members` + Google Places | 없음 |
| 피드 (탭) | 그룹별 서브탭으로 분리된 방문 기록 타임라인 (좋아요/댓글 포함) | `visits` + 신규 2개 테이블 | **있음** |
| 기록하기(+) (탭) | 새 방문 기록 작성 (저장 시 공유 대상 그룹 선택) | `visits` insert | 없음 |
| 추천 (탭) | 개인 취향 기반 새 라멘집 추천 (그룹 무관, 개인 전체 기록 기준) | `visits`(집계) + Google Places + Claude API | 없음 (신규 테이블 불필요) |
| 친구/그룹 관리 (지도 화면 상단 아이콘) | 내가 속한 그룹 목록, 그룹별 멤버 조회, 초대 코드 공유 | `group_members` + `users` + `groups` | 없음 |

## 3. 탭별 기능 명세서

### 3.1 지도 탭

#### 기능 1. 지도 초기화 및 위치 권한 처리
| 항목 | 내용 |
|---|---|
| 기능 | 앱 첫 실행 시 위치 권한 요청, 허용 시 현재 위치로 카메라 이동 |
| 입력 | 없음 (시스템 권한 응답) |
| 출력 | 현재 위치 기준 초기 카메라, 내 위치 마커 |
| 예외 처리 | • 권한 거부 → 기본 좌표(서울시청) 폴백 + 안내 배너 • GPS 꺼짐 → 동일 폴백 • 위치 조회 타임아웃(5초) → 폴백 후 재시도 버튼 |

#### 기능 2. 소속 그룹 방문 매장 마커 병합 표시 (폴라로이드 핀 + 그룹별 색상)
| 항목 | 내용 |
|---|---|
| 기능 | 내가 속한 모든 그룹(개인 그룹 포함)의 `visits` 기록이 있는 `ramen_shops`를 폴라로이드 사진 핀으로 병합 표시. 그룹마다 핀 테두리/워시테이프 색상을 다르게 해서 구분 |
| 입력 | 현재 유저가 속한 `group_id` 목록 (`group_members`로 조회) |
| 출력 | 마커 리스트 (매장명, 좌표, 최신 방문 썸네일/rating 요약, 소속 그룹명·색상) |
| 예외 처리 | • 아직 어떤 그룹에도 기록이 없음(가입 직후) → 빈 지도 + "첫 기록을 남겨보세요" 안내(정상 상태) • 네트워크 실패 → 캐시 데이터 노출 + 재시도 배너 • 동일 매장을 여러 그룹에서 각각 방문 → 그룹별로 별개 마커 유지(병합하지 않음, 색으로 구분되므로) • 동일 그룹 내 동일 매장 중복 방문 → 마커 1개 병합, 방문 횟수 뱃지 표시 |

> **그룹별 색상 배정**: `groups.id`를 해시해서 고정 팔레트(예: 8색) 중 하나에 매핑하는 방식으로 시작합니다. 그룹 테이블에 별도 컬럼을 추가하지 않아도 되고(스키마 변경 없음), 같은 그룹은 항상 같은 색이 나옵니다. 그룹이 많아져 색이 헷갈리는 시점이 오면 그때 `groups.color` 커스텀 컬럼 도입을 고려합니다.

#### 기능 3. 마커 탭 인터랙션

| 입력 | 탭한 마커의 `shop_id` |
| 출력 | 바텀시트: 매장명/주소, 그룹원별 방문 카드(썸네일, rating, memo, visited_at) |
| 예외 처리 | • 상세 조회 실패 → 바텀시트 내 에러+재시도 (지도는 유지) • 미디어 로드 실패 → placeholder 아이콘 대체 |

#### 기능 4. 친구/그룹 관리 진입 (상단 아이콘)
| 항목 | 내용 |
|---|---|
| 기능 | 지도 화면 상단 아이콘(`ti-users`) 탭 시 내가 속한 그룹 목록(개인 포함)으로 진입, 그룹 하나를 선택하면 해당 그룹의 멤버 목록/초대 화면으로 이동 |
| 입력 | 현재 유저가 속한 `group_id` 목록 |
| 출력 | 1단계: 그룹 리스트(그룹명, 색상, 멤버 수) / 2단계: 선택한 그룹의 멤버 리스트(닉네임, 아바타, 가입일) + 초대 코드 공유 버튼 |
| 예외 처리 | • 개인 그룹만 있는 경우(그룹 리스트에 1개뿐) → 바로 2단계로 스킵하고 "아직 함께하는 사람이 없어요" + 초대 유도 문구 • 초대 코드 공유 실패(클립보드 등) → 텍스트 노출 후 수동 복사 유도 |

#### 기능 5. 전국 라멘집 검색
| 항목 | 내용 |
|---|---|
| 기능 | 검색창에 가게 이름 입력 시 전국 라멘집을 Google Places로 검색, 지도에 결과 표시 |
| 입력 | 검색어(가게 이름), 현재 지도 카메라 중심 좌표(검색 위치 편향용) |
| 출력 | 검색 결과를 지도에 표시 — **이미 방문한 곳은 기존 폴라로이드 핀**, **아직 안 가본 곳은 심플 핀**(시각적으로 구분) |
| 예외 처리 | • 검색 결과 없음 → "검색 결과가 없어요" 안내 • Google API 실패/쿼터 초과 → 최근 검색 캐시 노출 + 재시도 안내 • 빈 검색어 → 검색 버튼 비활성화 • 네트워크 지연 → 로딩 인디케이터 |

> 안 가본 곳(심플 핀)을 탭하면 매장 상세 대신 바로 **"기록하기" 화면으로 진입**(매장 정보 자동 입력)하도록 연결합니다. 방문 기록이 없는 곳은 보여줄 방문 카드가 없기 때문입니다. 이 로직은 3.3 기록하기(+) 탭의 "매장 검색" 기능과 저장소(repository)를 공유하도록 설계합니다.

### 3.2 피드 탭 (신규)

| 예외 처리 | • 그룹이 개인 그룹 1개뿐 → 서브탭 자체를 숨기고 바로 피드 노출(불필요한 UI 제거) |

#### 기능 2. 피드 목록 조회
| 항목 | 내용 |

| 예외 처리 | • 선택된 그룹에 기록이 없음(개인 그룹 초기 상태 등) → 빈 피드 + "첫 기록을 남겨보세요" 안내(정상 상태) • 네트워크 실패 → 캐시 노출 + 재시도 배너 • 다음 페이지 로드 실패 → 현재까지 로드분 유지 + 하단 재시도 버튼 |

#### 기능 3. 좋아요 토글
| 항목 | 내용 |
|---|---|

| 입력 | `visit_id`, 현재 `user_id` |
| 출력 | 좋아요 상태 즉시 반영(optimistic update), 좋아요 수 갱신 |
| 예외 처리 | • 중복 좋아요 → DB unique 제약으로 차단, 클라이언트는 토글로만 인지 • 네트워크 실패 → optimistic update 롤백 + 토스트 안내 • 연속 탭(더블탭 등) → 요청 디바운스 처리 |

| 출력 | 댓글 리스트에 즉시 추가, 댓글 수 갱신 |

#### 기능 5. 댓글 목록 조회
| 항목 | 내용 |
|---|---|

| 출력 | 댓글 리스트 (작성자, 내용, 작성 시각) 오래된순 |
| 예외 처리 | • 조회 실패 → "댓글을 불러오지 못했어요" + 재시도 버튼 |

### 3.3 기록하기(+) 탭

#### 기능 1. 방문 기록 작성
| 항목 | 내용 |
|---|---|
| 기능 | 매장 검색/선택(또는 신규 등록) → 사진/영상 첨부 → 국물 종류/평점/메모 입력 → **공유 대상 그룹 선택**(개인/그룹A/그룹B 중 하나) → 저장 |
| 입력 | `shop_id`(또는 신규 매장의 `google_place_id`/이름/좌표), 미디어 파일, `broth_type`, `rating`, `memo`, 선택한 `group_id` |
| 출력 | `visits` row 생성 (`group_id` = 선택한 대상) → 지도/피드에 즉시 반영 |
| 예외 처리 | • 미디어 업로드 실패(용량 초과/네트워크) → 재시도, 실패해도 텍스트 정보는 임시 저장 • 매장 검색 결과 없음 → 신규 매장 등록 유도 • 필수값(평점, 공유 대상 등) 미입력 → 저장 버튼 비활성화 • 업로드 중 화면 이탈 → 업로드 완료까지 백그라운드 유지 또는 확인 다이얼로그 • 공유 대상 미선택 → 기본값으로 개인 그룹 선택되어 있음(명시적으로 바꿔야 그룹 공유) |

### 3.4 친구/그룹 관리 서브 화면 (지도 탭 기능 4에서 진입)

### 3.5 추천 탭 (신규, AI 기반)

> 방식: 규칙 기반 스코어링(후보 선정/순위) + Claude API(추천 이유 생성)의 하이브리드
> - 후보 선정/순위는 규칙 기반이 결정론적이고 비용이 안 들어 유리
> - "왜 추천했는지" 자연어 설명은 Claude API가 담당 (최종 후보 3~5곳에만 호출 → 비용 제한적)

#### 기능 1. 개인 취향 프로필 계산
| 항목 | 내용 |
|---|---|
| 기능 | 사용자 본인의 `visits` 기록을 집계해 국물 종류별 방문 빈도·평균 평점 산출 |
| 입력 | 현재 유저의 `user_id` |
| 출력 | 취향 프로필 (예: 돈코츠 40%·평균 4.5점, 쇼유 30%·평균 3.8점 …) |
| 예외 처리 | • 기록 3건 미만 → 추천 비활성화, "기록을 더 남기면 추천받을 수 있어요" 안내 • 특정 국물 종류에 편중 → 취향이 명확한 것으로 간주하고 진행 |

#### 기능 2. 후보 라멘집 검색 (Google Places)
| 항목 | 내용 |
|---|---|
| 기능 | 취향 상위 국물 종류를 키워드로 현재 위치 주변 라멘집 검색, 이미 방문한 곳 제외 |
| 입력 | 취향 프로필 상위 국물 종류, 현재 위치 좌표, 검색 반경 |
| 출력 | 후보 라멘집 리스트 (이름, 좌표, 평점, 리뷰 수) |
| 예외 처리 | • 검색 결과 없음 → 반경 확장 후 재검색, 그래도 없으면 안내 문구 • 위치 권한 없음 → 최근 방문 지역 평균 좌표로 폴백 • Google API 실패/쿼터 초과 → 캐시된 이전 추천 노출 |

#### 기능 3. 후보 스코어링 및 순위화 (규칙 기반)
| 항목 | 내용 |
|---|---|
| 기능 | 취향 프로필과 후보 매장의 속성(평점, 거리, 국물 키워드 일치도)으로 점수 계산 후 상위 N개 선정 |
| 입력 | 취향 프로필, 후보 리스트 |
| 출력 | 상위 3~5개 순위화된 후보 |
| 예외 처리 | • 스코어링 근거 부족(리뷰 정보 빈약) → 평점·거리 기준 기본 정렬로 폴백 |

#### 기능 4. Claude API 추천 이유 생성
| 항목 | 내용 |
|---|---|
| 기능 | 선정된 상위 후보에 대해 개인화된 추천 이유 한두 문장 생성 |
| 입력 | 취향 프로필 요약, 후보 매장 정보(이름/평점/특징) |
| 출력 | 각 후보 카드에 노출할 자연어 추천 문구 |
| 예외 처리 | • API 실패/타임아웃 → 이유 텍스트 없이 기본 카드(점수/거리)만 노출 • 응답 지연 5초 초과 → 기본 카드로 대체 • 동일 후보 반복 요청 → 캐싱(하루 1회)으로 비용 절감 |

#### 기술 설계 메모
- **Claude API 키는 클라이언트(Flutter 앱)에 넣지 않는다.** Supabase Edge Function을 통해 서버 사이드에서만 호출하고, 앱은 Edge Function만 호출하는 구조로 설계
- DB 스키마 변경 불필요 — 취향 프로필은 매번 `visits` 집계 쿼리로 계산 (트래픽 증가 시 캐싱 테이블 도입은 추후 최적화 과제)
- 국물 종류 키워드 기반 Google 검색은 지역별 결과 편차가 있을 수 있어, 초기 버전은 "그럴듯한 후보" 제공에 집중하고 사용자 피드백(예: '마음에 안 듦' 버튼)으로 보정하는 것은 다음 단계 개선 과제로 분리

## 4. DB 스키마 변경사항 (피드 좋아요/댓글 + 다중 그룹 지원)

기존 5개 테이블 구조는 유지하며, 아래 내용을 추가합니다.

### 4.1 피드 좋아요/댓글 (신규 테이블 2개)

```sql
create table public.visit_likes (
  id uuid primary key default gen_random_uuid(),
  visit_id uuid references public.visits(id) on delete cascade,
  user_id uuid references public.users(id) on delete cascade,
  created_at timestamptz default now(),
  unique (visit_id, user_id)
);

create table public.visit_comments (
  id uuid primary key default gen_random_uuid(),
  visit_id uuid references public.visits(id) on delete cascade,
  user_id uuid references public.users(id) on delete cascade,
  content text not null,
  created_at timestamptz default now()
);
```

### 4.2 다중 그룹 지원 (신규 컬럼 없음)

전역 활성 그룹 상태를 두지 않기로 했으므로(1.1절 참고), 별도 컬럼 추가가 필요 없습니다.

- 신규 유저 가입 시 서버(트리거 또는 Edge Function)에서 **개인 그룹 자동 생성 → `group_members`에 본인 추가**하는 것으로 끝
- 초대코드로 새 그룹에 합류할 때는 `group_members`에 row만 추가(기존 멤버십 삭제·변경 없음)
- `group_members`는 원래도 그룹-유저 간 조인 테이블이라 다중 소속 자체는 스키마 변경 없이 이미 지원됨 — **다만 `group_members(group_id, user_id)`에 UNIQUE 제약이 있는지 확인 필요** (같은 그룹에 중복 가입 방지용, 없으면 추가 권장)

```sql
alter table public.group_members
  add constraint group_members_unique unique (group_id, user_id);
```

### ⚠️ 기존 트러블슈팅 기록 기반 필수 체크리스트

CLAUDE.md의 6번 항목(겪었던 트러블슈팅)에 따라, 아래를 **반드시 마이그레이션에 포함**해야 합니다.

- `GRANT SELECT, INSERT, UPDATE, DELETE ON public.visit_likes TO authenticated;`
- `GRANT SELECT, INSERT, UPDATE, DELETE ON public.visit_comments TO authenticated;`
- RLS 정책
  - SELECT: 같은 그룹 멤버만 조회 가능 (`visits` → `group_id` → `group_members` 조인 확인)
  - INSERT: 본인 `user_id`로만 생성 가능 (좋아요 중복은 `unique` 제약이 DB 레벨에서 차단)
  - DELETE(좋아요 취소): 본인이 작성한 좋아요만 삭제 가능
  - UPDATE/DELETE(댓글): 본인이 작성한 댓글만 수정/삭제 가능

## 5. 공통 모듈 설계 — 라멘집 검색 (Google Places 연동)

### 5.1 공유 이유

동일한 "라멘집 검색" 로직이 세 곳에서 필요합니다.

- 지도 탭 기능 5 (전국 검색)
- 추천 탭 기능 2 (후보 검색)
- 기록하기(+) 탭 기능 1 (매장 검색/신규 등록)

각각 따로 구현하면 API 호출 방식·결과 파싱·에러 처리가 제각각이 되어 유지보수가 어려워지므로, **단일 저장소(Repository)로 통합**합니다.

### 5.2 모듈 인터페이스 설계

| 항목 | 내용 |
|---|---|
| 모듈명 | `RamenShopSearchRepository` (가칭) |
| 입력 | 검색어(선택), 위치 좌표(위치 편향), 검색 반경(선택) |
| 출력 | 정규화된 결과 리스트 — 각 항목에 `isRegistered`(우리 `ramen_shops`에 이미 있는지), `visitCount`(그룹 방문 횟수, 0이면 미방문) 플래그 포함 |
| 예외 처리 | • Google API 실패/쿼터 초과 → 캐시된 결과 반환, 캐시도 없으면 빈 리스트+안내 • 검색어 없이 호출 시 위치 기반 주변 검색으로 폴백 • 중복 호출 방지를 위한 디바운스(예: 300ms) |

### 5.3 사용처별 분기 로직

- **지도 탭**: `isRegistered && visitCount > 0` → 폴라로이드 핀 / 그 외 → 심플 핀
- **추천 탭**: `visitCount > 0`인 곳(이미 가본 곳)은 후보에서 제외
- **기록하기(+) 탭**: `isRegistered = false`인 곳을 선택하면 저장 시 `ramen_shops`에 신규 upsert (`google_place_id` 기준)

### 5.4 비용/성능 고려사항

- Google Places API는 호출당 과금되므로 **검색어+위치 버킷 단위 캐싱**을 클라이언트 메모리(짧은 TTL)로 우선 적용. 서버 사이드 캐시 테이블은 트래픽이 늘었을 때 최적화 과제로 남김 (지금은 스키마 추가 안 함 — YAGNI 원칙)
- 지도 탭에서 카메라를 움직일 때마다 재검색하면 호출량이 급증하므로, **카메라 이동이 멈추고 일정 시간(예: 500ms) 후에만 재검색**하도록 제한
