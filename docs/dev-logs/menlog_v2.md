## 1. 개요

- 기존에는 "홈(지도 단일 화면) → 기록 폼 → 초대 화면"의 단선적 흐름이었으나, 이번 기획부터 **하단 탭 4개(지도/피드/기록하기/추천) + 지도 화면 상단 아이콘(친구/그룹 관리)** 구조로 전환
- 디자인은 PDF 1a 스타일(세리프 타이틀, 폴라로이드 사진 핀) 채택 — 기존 확정 디자인 토큰(Primary `#C8600A`, Georgia 세리프 워드마크)과 정합성 확인 완료
- **탭바 재배치 판단 근거**: 지도·피드·기록하기·추천은 반복 사용 빈도가 높은 핵심 동선인 반면, 친구(그룹 관리)는 초대 시 1회성으로 쓰이고 이후엔 저빈도로 사용됨. 저빈도 기능에 탭 슬롯을 배정하면 핵심 탭들의 터치 영역만 좁아지는 손해가 있어, 지도 화면 상단 아이콘(그룹원 아바타/`ti-users`)으로 배치
- '친구' 기능은 신규 개념이 아니라 **기존 `groups`/`group_members`의 UI 라벨링 변경**이며, 배치만 탭→아이콘으로 이동
- '피드' 탭은 신규 기능이며, 좋아요/댓글 상호작용을 포함하므로 **스키마 확장 필요**

## 1.1 그룹 모델 변경 — "혼자서도 쓸 수 있는 앱" 반영

기존에는 "그룹에 소속돼야만 앱을 쓸 수 있다"는 전제였으나, 혼자 쓰는 유저도 지원하기로 결정했습니다.

- **가입 즉시 "개인 그룹" 자동 생성**: 신규 유저는 닉네임 설정 직후 백그라운드에서 본인만 속한 그룹이 자동 생성됩니다. 화면상 별도 "그룹 만들기" 단계는 없습니다 — 사용자에게는 그냥 "내 기록"으로 보입니다.
- **다중 그룹 모델 채택**: 이후 여러 친구 그룹의 초대코드를 받아 동시에 여러 그룹에 소속될 수 있습니다 (개인 그룹 + 친구 그룹 N개). 그룹 하나를 폐기하거나 갈아타는 개념이 아닙니다.
- **"전역 활성 그룹" 상태를 두지 않기로 결정** (그룹 전환 UI 방식 기각): 화면마다 그룹을 다루는 방식이 다릅니다.
  - **지도**: 내가 속한 모든 그룹(개인 포함)의 기록을 **한 지도에 병합**해서 보여주고, 그룹마다 다른 마커 색으로 구분 — 전환 없이 항상 전체를 봄
  - **피드**: 그룹마다 성격이 다른 사교 공간이라 병합하지 않고, **그룹별 서브탭**으로 분리해서 보여줌 (선택 상태는 로컬 UI 상태로만 기억, DB 저장 불필요)
  - **기록하기(+)**: 저장 시점마다 "이 기록을 어디에 남길지"(개인/그룹A/그룹B)를 명시적으로 선택 — 전역 상태에 의존하지 않음
  - **추천**: 그룹과 무관하게 유저 개인의 전체 방문 기록(모든 그룹 합산)을 취향 계산에 사용 — 기존 설계 그대로 유지
- 이 방식 덕분에 `users.active_group_id` 같은 전역 상태 컬럼이 **필요 없어졌습니다.**

## 3. 탭별 기능 명세서

### 3.1 지도 탭 (행정구역 정복맵 방식으로 전환)

> **아키텍처 전환**: 실제 GPS 기반 Google Maps를 포기하고, 시/군/구 단위 행정구역 폴리곤을 채색하는 커스텀 지도로 전환합니다. Google Maps SDK(`GoogleMap` 위젯, 네이티브 API 키 설정, `map_style.json`)는 더 이상 필요 없습니다. 단, **Google Places API는 계속 사용합니다** — 매장의 실제 이름/주소/좌표를 식별하는 용도(기록하기, 추천 탭, 지역 매칭)로는 여전히 필요합니다.

#### 기능 1. 행정구역 정복맵 렌더링
| 항목 | 내용 |
|---|---|
| 기능 | 시/군/구 단위 폴리곤을 그려서, 선택된 그룹이 방문 기록을 남긴 지역은 채색(정복), 안 남긴 지역은 빈 상태로 표시 |
| 입력 | 선택된 그룹의 `group_id`, 해당 그룹의 `visits`가 있는 지역 목록 |
| 출력 | 전국 지도(시/군/구 폴리곤), 정복 지역은 색칠 + 대표 사진(가장 최근 방문 매장 사진, 해당 지역 모양으로 클리핑) + **지역 내 서로 다른 매장이 2개 이상이면 사진 위에 "+N" 뱃지 표시**(N = 매장 수 − 1), 미정복 지역은 빈 폴리곤 |
| 예외 처리 | • 선택된 그룹에 기록이 하나도 없음 → 전체 미정복 상태 + "첫 기록을 남겨보세요" 안내(정상 상태) • 경계 데이터 로드 실패 → 이전 캐시 표시 + 재시도 배너 • 매장 주소가 행정구역 매칭에 실패(해외 주소, 데이터 누락 등) → 해당 매장은 지도에서 제외하고 로그만 남김(사용자에게 에러 노출 안 함) • 같은 매장을 여러 번 방문 → 매장 수 집계에는 1개로만 카운트(방문 횟수가 아니라 서로 다른 매장 수 기준) |

> **색상 스펙(참고 스크린샷은 다크 테마였지만, 크래프트지 웜톤 팔레트로 대체)**
> - 배경(바다·미정복 여백): `MenlogColors.background` (#EDE0CC)
> - 미정복 지역 폴리곤: `MenlogColors.surface` (#F5D9B8), 테두리 `MenlogColors.text`와 저채도 톤
> - 정복 지역 폴리곤(사진 없는 배경 부분): `MenlogColors.primary` (#C8600A) 계열
> - 지역명 라벨 텍스트: `MenlogColors.text` (#6B3E26)
> - "+N" 뱃지: `MenlogColors.dark` 배경 + 흰 텍스트 (기존 친구 아이콘 뱃지와 동일 톤)

> **기술 메모**: 행정구역 경계는 공공데이터포털의 SIG(시군구) 경계 GeoJSON을 기준 데이터로 사용합니다. 매장을 어느 지역에 매칭할지는 좌표 기반 point-in-polygon 계산 대신, **Google Places가 반환하는 주소 구성요소(시/군/구 명칭)를 그대로 문자열 매칭**하는 방식이 훨씬 간단하고 안정적입니다 — 이미 기록하기 탭에서 매장을 검색할 때 주소 정보를 받아오므로 별도 계산 없이 재사용 가능합니다.

#### 기능 2. 그룹 선택 (화면 내 로컬 상태)
| 항목 | 내용 |
|---|---|
| 기능 | 지도 화면 안에서 "어느 그룹 기준으로 정복 상태를 볼지" 선택 (피드 탭의 그룹 서브탭과 동일한 패턴 재사용) |
| 입력 | 현재 유저가 속한 `group_id` 목록 |
| 출력 | 그룹 선택 UI(서브탭 또는 드롭다운), 선택 상태는 로컬 UI 상태로만 기억(DB 저장 불필요) |
| 예외 처리 | • 개인 그룹 1개뿐 → 선택 UI 자체를 숨기고 바로 그 그룹 기준으로 표시 |

> 이전에 "전역 활성 그룹 상태를 두지 않는다"고 결정했던 원칙(1.1절)을 그대로 따릅니다. 지도 탭도 전역 상태 대신, 피드 탭처럼 화면 안에서만 쓰는 로컬 선택 상태로 처리합니다.

#### 기능 3. 지역 탭 인터랙션
| 항목 | 내용 |
|---|---|
| 기능 | 정복된 지역 폴리곤 탭 시 해당 지역의 방문 기록 요약 바텀시트 표시 |
| 입력 | 탭한 지역 코드, 선택된 `group_id` |
| 출력 | 바텀시트: 지역명, 그 지역 내 매장별 방문 카드(썸네일, rating, memo, visited_at) |
| 예외 처리 | • 미정복 지역 탭 → 바텀시트 대신 "아직 기록이 없어요, 기록하러 가볼까요?" + 기록하기 진입 유도 • 상세 조회 실패 → 바텀시트 내 에러+재시도 |

#### 기능 4. 확대/드릴다운 인터랙션
| 항목 | 내용 |
|---|---|
| 기능 | 전국 단위 지도에서 특정 시/도를 탭하면 그 안의 시/군/구 단위로 확대해서 보여줌 (스크린샷 예시처럼) |
| 입력 | 탭한 시/도 코드 |
| 출력 | 확대된 시/군/구 폴리곤 화면, 뒤로가기로 전국 단위 복귀 |
| 예외 처리 | • 확대 중 데이터 로딩 지연 → 스켈레톤/로딩 인디케이터 |

#### 기능 5. 친구/그룹 관리 진입 (상단 아이콘)
| 항목 | 내용 |
|---|---|
| 기능 | 지도 화면 상단 아이콘(`ti-users`) 탭 시 내가 속한 그룹 목록(개인 포함)으로 진입, 그룹 하나를 선택하면 해당 그룹의 멤버 목록/초대 화면으로 이동 |
| 입력 | 현재 유저가 속한 `group_id` 목록 |
| 출력 | 1단계: 그룹 리스트(그룹명, 색상, 멤버 수) / 2단계: 선택한 그룹의 멤버 리스트(닉네임, 아바타, 가입일) + 초대 코드 공유 버튼 |
| 예외 처리 | • 개인 그룹만 있는 경우(그룹 리스트에 1개뿐) → 바로 2단계로 스킵하고 "아직 함께하는 사람이 없어요" + 초대 유도 문구 • 초대 코드 공유 실패(클립보드 등) → 텍스트 노출 후 수동 복사 유도 |

#### 기능 6. 전국 라멘집 검색
| 항목 | 내용 |
|---|---|
| 기능 | 검색창에 가게 이름 입력 시 전국 라멘집을 Google Places로 검색 |
| 입력 | 검색어(가게 이름) |
| 출력 | 검색된 매장이 속한 지역으로 지도 확대 이동 + 해당 지역 하이라이트. 이미 방문한 매장이면 그 지역의 방문 카드 바텀시트로 바로 연결 |
| 예외 처리 | • 검색 결과 없음 → "검색 결과가 없어요" 안내 • Google API 실패/쿼터 초과 → 최근 검색 캐시 노출 + 재시도 안내 • 빈 검색어 → 검색 버튼 비활성화 |

> 안 가본 매장을 검색해서 찾은 경우, 바텀시트 대신 바로 **"기록하기" 화면으로 진입**(매장 정보 자동 입력)하도록 연결합니다. 이 로직은 3.3 기록하기(+) 탭의 "매장 검색" 기능과 저장소(repository)를 공유하도록 설계합니다.

### 3.2 피드 탭 (신규)

#### 기능 1. 그룹 서브탭 선택
| 항목 | 내용 |
|---|---|
| 기능 | 피드 상단에 내가 속한 그룹(개인 포함)을 서브탭/칩 형태로 나열, 선택한 그룹의 피드만 노출 |
| 입력 | 현재 유저가 속한 `group_id` 목록 |
| 출력 | 그룹별 서브탭 UI, 마지막으로 본 그룹을 다음 진입 시에도 기억(로컬 상태, DB 저장 불필요) |
| 예외 처리 | • 그룹이 개인 그룹 1개뿐 → 서브탭 자체를 숨기고 바로 피드 노출(불필요한 UI 제거) |

#### 기능 2. 피드 목록 조회
| 항목 | 내용 |
|---|---|
| 기능 | 선택된 그룹의 모든 방문 기록을 최신순으로 노출 |
| 입력 | 선택된 `group_id`, 페이지네이션 커서 |
| 출력 | 방문 카드 리스트 (작성자 닉네임/아바타, 매장명, 사진, rating, memo, 좋아요 수, 댓글 수, visited_at) |
| 예외 처리 | • 선택된 그룹에 기록이 없음(개인 그룹 초기 상태 등) → 빈 피드 + "첫 기록을 남겨보세요" 안내(정상 상태) • 네트워크 실패 → 캐시 노출 + 재시도 배너 • 다음 페이지 로드 실패 → 현재까지 로드분 유지 + 하단 재시도 버튼 |

#### 기능 3. 좋아요 토글
| 항목 | 내용 |
|---|---|
| 기능 | 방문 카드에 좋아요 등록/취소 |
| 입력 | `visit_id`, 현재 `user_id` |
| 출력 | 좋아요 상태 즉시 반영(optimistic update), 좋아요 수 갱신 |

## 4. DB 스키마 변경사항 (피드 좋아요/댓글 + 다중 그룹 지원)

기존 5개 테이블 구조는 유지하며, 아래 내용을 추가합니다.

### 4.1 피드 좋아요/댓글 (신규 테이블 2개)

```sql
create table public.visit_likes (
  id uuid primary key default gen_random_uuid(),
  visit_id uuid references public.visits(id) on delete cascade,
  user_id uuid references public.users(id) on delete cascade,
  created_at timestamptz default now(),
  unique (visit_id, user_id)
);

create table public.visit_comments (
  id uuid primary key default gen_random_uuid(),
  visit_id uuid references public.visits(id) on delete cascade,
  user_id uuid references public.users(id) on delete cascade,
  content text not null,
  created_at timestamptz default now()
);
```

### 4.2 다중 그룹 지원 (신규 컬럼 없음)

전역 활성 그룹 상태를 두지 않기로 했으므로(1.1절 참고), 별도 컬럼 추가가 필요 없습니다.

- 신규 유저 가입 시 서버(트리거 또는 Edge Function)에서 **개인 그룹 자동 생성 → `group_members`에 본인 추가**하는 것으로 끝
- 초대코드로 새 그룹에 합류할 때는 `group_members`에 row만 추가(기존 멤버십 삭제·변경 없음)
- `group_members`는 원래도 그룹-유저 간 조인 테이블이라 다중 소속 자체는 스키마 변경 없이 이미 지원됨 — **다만 `group_members(group_id, user_id)`에 UNIQUE 제약이 있는지 확인 필요** (같은 그룹에 중복 가입 방지용, 없으면 추가 권장)

```sql
alter table public.group_members
  add constraint group_members_unique unique (group_id, user_id);
```

### ⚠️ 기존 트러블슈팅 기록 기반 필수 체크리스트

CLAUDE.md의 6번 항목(겪었던 트러블슈팅)에 따라, 아래를 **반드시 마이그레이션에 포함**해야 합니다.

- `GRANT SELECT, INSERT, UPDATE, DELETE ON public.visit_likes TO authenticated;`
- `GRANT SELECT, INSERT, UPDATE, DELETE ON public.visit_comments TO authenticated;`
- RLS 정책
  - SELECT: 같은 그룹 멤버만 조회 가능 (`visits` → `group_id` → `group_members` 조인 확인)
  - INSERT: 본인 `user_id`로만 생성 가능 (좋아요 중복은 `unique` 제약이 DB 레벨에서 차단)
  - DELETE(좋아요 취소): 본인이 작성한 좋아요만 삭제 가능
  - UPDATE/DELETE(댓글): 본인이 작성한 댓글만 수정/삭제 가능

## 5. 공통 모듈 설계 — 라멘집 검색 (Google Places 연동)

### 5.1 공유 이유

동일한 "라멘집 검색" 로직이 세 곳에서 필요합니다.

- 지도 탭 기능 6 (전국 검색)
- 추천 탭 기능 2 (후보 검색)
- 기록하기(+) 탭 기능 1 (매장 검색/신규 등록)

각각 따로 구현하면 API 호출 방식·결과 파싱·에러 처리가 제각각이 되어 유지보수가 어려워지므로, **단일 저장소(Repository)로 통합**합니다.

### 5.2 모듈 인터페이스 설계

| 항목 | 내용 |
|---|---|
| 모듈명 | `RamenShopSearchRepository` (가칭) |
| 입력 | 검색어(선택), 위치 좌표(위치 편향), 검색 반경(선택) |
| 출력 | 정규화된 결과 리스트 — 각 항목에 `isRegistered`(우리 `ramen_shops`에 이미 있는지), `visitCount`(그룹 방문 횟수, 0이면 미방문), **`regionCode`(Google Places 주소 구성요소에서 추출한 시/군/구 코드 — 지도 탭 정복맵 매칭용)** 플래그 포함 |
| 예외 처리 | • Google API 실패/쿼터 초과 → 캐시된 결과 반환, 캐시도 없으면 빈 리스트+안내 • 검색어 없이 호출 시 위치 기반 주변 검색으로 폴백 • 중복 호출 방지를 위한 디바운스(예: 300ms) • 주소 구성요소에 시/군/구 정보가 없는 경우(해외 등) → `regionCode`는 null로 두고 정복맵에서는 제외 |

### 5.3 사용처별 분기 로직

- **지도 탭**: `isRegistered && visitCount > 0` → 폴라로이드 핀 / 그 외 → 심플 핀
- **추천 탭**: `visitCount > 0`인 곳(이미 가본 곳)은 후보에서 제외
- **기록하기(+) 탭**: `isRegistered = false`인 곳을 선택하면 저장 시 `ramen_shops`에 신규 upsert (`google_place_id` 기준)

- Google Places API는 호출당 과금되므로 **검색어+위치 버킷 단위 캐싱**을 클라이언트 메모리(짧은 TTL)로 우선 적용. 서버 사이드 캐시 테이블은 트래픽이 늘었을 때 최적화 과제로 남김 (지금은 스키마 추가 안 함 — YAGNI 원칙)
